## 7.5 Residual Risk Summary

Two categories of residual risk remain unaddressed after this assessment pass:

**LLM integration residual risk:** TrunkNuX mediates between user-controlled project data and the Claude API. The `[VERIFY]` marker system (R-DISC-02) is the primary control for catching hallucinated or injected content, but it relies entirely on human review. There is no automated semantic check to detect prompt injection attempts in test fixture data, nor any mechanism to detect when LLM output contains information that should not be written to disk. The `--max-spend` cost gate (R-DISC-07) limits financial exposure but does not address data-exfiltration risk. This risk is LOW at v0.2.2 because the CLI is used by a single operator, but will need re-evaluation before any multi-tenant or team deployment.

**Supply chain residual risk:** TrunkNuX's audit PDF rendering relies on `puppeteer-core` (optional) and `@anthropic-ai/sdk` (required). Neither dependency is verified beyond `package-lock.json` pin and `npm audit`. A compromised version of either package could silently modify audit artifacts or exfiltrate API keys. Sigstore/cosign verification and SBOM generation are planned for v0.3 but are not yet in place. Until then, consumers should pin to verified release tags.
