# TestNUX v0.2 — RTM + SCA Generators

TestNUX v0.2 adds two new commands that cover steps 4 and 5 of the 8-step
regulator-evidence chain:

```
3. Testing-log (test plans + Playwright + reports)    ← v0.1
4. TRACEABILITY.md (R-XX → sprint → code → test)     ← v0.2 NEW: testnux rtm
5. Security Control Assessments (NIST 800-53A SCA)   ← v0.2 NEW: testnux sca
```

---

## testnux rtm

Generates `requirements/TRACEABILITY.md` from your project's living documentation.

### Quick start

```bash
npx testnux rtm
```

### What it does

1. Parses `requirements/REQUIREMENTS.md` for R-IDs and titles.
   Accepts both `## R-01 Title` and `## R-01 — Title` heading styles.

2. Walks every `sprint-log/<DATE>_<feature>/SPRINT_SUMMARY.md` and extracts
   R-IDs mentioned in shipped-work lines.

3. Greps the codebase (default: `src/**` and `app/**`) for inline `// R-XX`
   comment annotations.

4. Greps test files (default: `**/*.spec.ts`, `testing-log/**/*.md`) for
   `describe('R-XX', ...)` blocks and `// R-XX` markers.

5. Reads `requirements/MASTER_BACKLOG.md` for open items per R-ID (if present).

6. Writes `requirements/TRACEABILITY.md` — one row per R-ID, columns:
   STATUS / SPRINT / CODE / TESTS / BACKLOG / NOTES.

### Flags

| Flag | Description |
|------|-------------|
| `--dry-run` | Print the generated markdown to stdout; do not write the file |
| `--strict` | Exit 1 if any R-ID has no code or test evidence |
| `--config <path>` | Path to `testnux.config.mjs` for glob overrides |

### Exit codes

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Generic error (or strict-mode failure) |
| 2 | `requirements/REQUIREMENTS.md` not found |
| 3 | Parse error in requirements file |

### Glob overrides via config

Create a `testnux.config.mjs` in your project root:

```js
export default {
  rtm: {
    requirementsFile: 'docs/requirements/REQUIREMENTS.md',
    traceabilityFile: 'docs/requirements/TRACEABILITY.md',
    codeGlobs: ['lib/**/*.ts', 'app/**/*.tsx'],
    testGlobs: ['tests/**/*.spec.ts', 'e2e/**/*.ts'],
  },
};
```

## testnux sca

Generates per-surface Security Control Assessments (SCAs) as markdown documents
with 8 canonical sections. Three subcommands:

### sca init

```bash
npx testnux sca init <surface> [--industry general|fintech|healthcare]
```

Scaffolds `requirements/validations/<surface>/v1.0_<DATE>.md` from the built-in
8-section SCA template. The `--industry` flag selects which standards profile to
apply (controls loaded from `src/config/industry-standards/<industry>.json`).

**Example:**

```bash
npx testnux sca init login --industry general
# Creates: requirements/validations/login/v1.0_2026-04-26.md
```

### sca generate

```bash
npx testnux sca generate <surface>
```

Fills the per-control evidence rows in the latest SCA file for `<surface>`.
Evidence is drawn from:

- R-ID mappings (cross-referenced from REQUIREMENTS.md)
- Sprint evidence (from sprint-log summaries)
- Code annotations (`// R-XX` inline markers)
- Test plans (from testing-log frontmatter + test specs)

Cells that cannot be auto-filled are marked `[VERIFY]` — see below.

### sca pdf

```bash
npx testnux sca pdf <surface>
```

Renders the latest SCA for `<surface>` to a PDF. Requires `puppeteer-core`
(optional dependency):

```bash
npm install puppeteer-core
```

Set `CHROME_PATH` or `PUPPETEER_EXECUTABLE_PATH` environment variables if
Chrome is not auto-detected.

If `puppeteer-core` is not installed, the command prints instructions for
rendering manually with `pandoc`.

### Flags (all sca subcommands)

| Flag | Description |
|------|-------------|
| `--dry-run` | Print output to stdout without writing |
| `--config <path>` | Path to `testnux.config.mjs` |
| `--standards-version <ver>` | Standards snapshot version in frontmatter (default: `1.0.0`) |
| `--industry <name>` | `init` only — standards profile |

## Marker convention — how human edits survive regeneration

Both `rtm` and `sca generate` use HTML comment markers so hand-written content
is preserved across runs.

### RTM row markers

Each row in `TRACEABILITY.md` is wrapped:

```markdown
<!-- testnux:row R-01 begin -->
| R-01 | Login page renders | DONE | 2026-01-10_login | ... | ... | Your note here |
<!-- testnux:row R-01 end -->
```

When you re-run `testnux rtm`, the generator:
- Rebuilds all columns from fresh evidence (STATUS, SPRINT, CODE, TESTS, BACKLOG)
- Preserves whatever text is in the **Notes** column (last column) inside the markers

**Do not edit columns other than Notes by hand.** They are overwritten on every run.

### SCA section markers

Human-edited sections in SCA documents are wrapped:

```markdown
<!-- testnux:section exec-summary begin -->
## 1. Executive Summary

... your edited content here ...

<!-- testnux:section exec-summary end -->
```

Protected sections: `exec-summary`, `methodology`, `threat-matrix`, `declined`,
`open-items`, `sign-off`.

When you re-run `testnux sca generate`, the generator rebuilds the Per-Control
Inventory and Standards Alignment Matrix from fresh evidence, while preserving
any hand-edits to those six sections.

### SCA control-level notes

Inside Section 3 (Per-Control Inventory), each row also has a row marker:

```markdown
<!-- testnux:row ctrl-ASVS-V2.1.1 begin -->
| ASVS-V2.1.1 | ... | Your operational note here |
<!-- testnux:row ctrl-ASVS-V2.1.1 end -->
```

The **Operational Notes** column (last column) survives regeneration; all other
columns are overwritten with fresh evidence.

## What [VERIFY] means

`[VERIFY]` marks any cell that the generator cannot fill automatically because
it would require:

- **Narrative judgment** — e.g., overall risk posture, executive summary bullets
- **Cross-reference research** — e.g., mapping attack scenarios to specific tests
- **Human attestation** — e.g., sign-off names, dates, declined-by-design rationale

`[VERIFY]` is an explicit uncertainty signal, not a placeholder to delete. Before
signing off on an SCA, every `[VERIFY]` should be resolved:

1. If you can fill it factually — replace `[VERIFY]` with the correct value.
2. If it needs LLM assistance — v0.2 will wire the Claude API to auto-fill these
   cells using the evidence context (R-ID + test results + control spec).
3. If it is genuinely not applicable — replace with `N/A` and add a note.

The count of remaining `[VERIFY]` markers is a quick readiness signal: an SCA
with zero `[VERIFY]` cells is ready for auditor review.

## Industry standards profiles

| Profile | Controls included |
|---------|-------------------|
| `general` | OWASP ASVS 4.0 (authentication, session, access control, input validation, data protection, TLS, API) + WCAG 2.2 AA |
| `fintech` | All of `general` + NIST 800-63B, NYDFS 23 NYCRR 500, PSD2 SCA, PCI DSS, FFIEC *(planned)* |
| `healthcare` | All of `general` + HIPAA Security Rule, HITECH, NIST 800-66 *(planned)* |

Add custom standards by creating `src/config/industry-standards/<name>.json`
following the same schema as `general.json`.

## The 8-section SCA structure

| # | Section | Auto-filled? | Human-editable? |
|---|---------|-------------|-----------------|
| 1 | Executive Summary | Partial (counts) | Yes — narrative |
| 2 | Methodology | Yes | Yes — override |
| 3 | Per-Control Inventory | Yes | Notes column only |
| 4 | Standards Alignment Matrix | Yes | No |
| 5 | Threat Coverage Matrix | No | Yes |
| 6 | Declined-by-Design | No | Yes |
| 7 | Open Items & Known Gaps | No | Yes |
| 8 | Sign-Off | No | Yes |

## Shared library

Both commands share `src/lib/parser.mjs` (pure parsing functions) and
`src/lib/graph.mjs` (in-memory evidence graph). These are the stable API
surface for programmatic use:

```js
import { parseRequirements, parseSprintSummary, parseTestPlan, parseCodeAnnotations } from 'testnux/lib/parser';
import { buildGraph } from 'testnux/lib/graph';

const requirements = parseRequirements(fs.readFileSync('requirements/REQUIREMENTS.md', 'utf-8'));
const graph = buildGraph({ requirements, sprints: [], tests: [], code: new Map() });
console.log(graph.coverageStats());
// { totalRIds: 42, withTests: 38, withCode: 35, withSprint: 40, coverage: 95 }
```
