# LLM Agents — TestNUX v0.2

TestNUX v0.2 ships four LLM agents that accelerate the test-documentation workflow.
Each command is a **v0.1 stub** — it runs today and prints exactly what v0.2 will do,
so you can plan integration and set budget expectations before the feature ships.

---

## The 4-Agent Pipeline

```
discover <url>
    ↓  scenarios.md  (Given/When/Then TCs, P0/P1/P2 tags)
plan <slug>
    ↓  test-plan.md  (TestNUX schema, R-IDs, steps, evidence checklist)
codify <slug>
    ↓  spec.ts       (Playwright TypeScript, getByRole(), afterEach screenshots)
enrich <slug>
    ↓  appends to test-plan.md (design, QA, graph passes — append-only)
```

Each stage is independent. You can run any stage manually and pick up from there with
the LLM agent. The agents read the output of the previous stage — they do not require
the previous stage to have been run by an agent.

### `testnux discover <url>` — v0.2 ALPHA (wired to Claude API)

Fetches the page at `<url>`, extracts a lightweight DOM summary (title, headings,
interactive elements, ARIA labels) using regex-based extraction (no heavy HTML parser
dependency), and sends the snapshot to Claude with a structured prompt that generates
Given/When/Then test scenarios for every interactive element. Output: `scenarios.md`.

Coverage per element: happy path, boundary values, invalid inputs, empty states,
permission edge cases, keyboard navigation (WCAG), security injection (P1 for forms).
Every LLM-generated TC ends with a `> [VERIFY]` blockquote — automatic per convention.

**Try it:**

```bash
export CLAUDE_API_KEY=sk-ant-...
npm install @anthropic-ai/sdk

testnux discover https://example.com/login --dry-run    # preview prompt + cost estimate
testnux discover https://example.com/login              # run for real → scenarios.md
testnux discover https://example.com/login --output ./testing-log/2026-04-26_login/
testnux discover https://example.com/login --model claude-haiku-4-5 --max-tokens 4000
```

**Flags:**

| Flag | Default | Description |
|---|---|---|
| `--output <dir>` | `.` | Directory where `scenarios.md` is written |
| `--model <model>` | `claude-sonnet-4-6` | Claude model (any Anthropic Messages API model) |
| `--max-tokens <n>` | `8000` | Max response tokens |
| `--dry-run` | off | Print prompt + cost estimate, skip API call and file write |
| `--slug <slug>` | derived from URL path | Override the `slug:` field in YAML frontmatter |
| `--json` | off | Emit all output as newline-delimited JSON records |

### `testnux plan <slug>`

Reads `<slug>-scenarios.md` and converts each TC into a fully structured TestNUX
`test-plan.md` entry: YAML frontmatter (tc_prefix, r_ids, standards), step-by-step
instructions, evidence checklist. Optionally takes a `--url` for a live DOM snapshot
to improve R-ID mapping accuracy.

### `testnux codify <slug>`

Reads `test-plan.md` and generates a typed Playwright TypeScript `spec.ts` following
strict conventions: `getByRole()`/`getByLabel()` selectors, no `waitForTimeout()`,
`afterEach()` screenshot hook for FAIL evidence, P0 tests in serial describe block.
Every assertion gets a `// [VERIFY]` comment.

### `testnux enrich <slug>`

Runs three parallel enrichment passes against an existing `test-plan.md` and appends
suggested TCs (APPEND-ONLY — never modifies human-curated content above the marker):

| Pass | Prefix | Focuses on |
|---|---|---|
| Design review | TC-A01, TC-D01 | WCAG 2.2 AA, APCA contrast, touch targets, focus indicators, reflow |
| QA structural | TC-E01 | Boundary values, CRUD matrix, concurrency, permission matrix, injection |
| Graph context | TC-I01 | Cross-surface integration, prerequisites, data flows, shared state |

### `testnux batch-plan --pages "login,register,dashboard"`

Dispatches parallel LLM agents for multi-page planning. Uses the replacement-agent
pattern for cost control: each sub-agent handles one batch of pages in a fresh context
window, preventing unbounded token accumulation.

## APPEND-ONLY Discipline

The `enrich` agent operates in strict append-only mode. It will **never** modify content
above the `<!-- testnux:enrich:start -->` marker. This protects every human-curated
test case and handwritten annotation from accidental overwrite.

**Section markers in `test-plan.md`:**

```markdown
<!-- testnux:enrich:start -->
<!-- DO NOT MODIFY ABOVE THIS LINE — human-curated content -->

## TC-A01 — Color contrast at 4.5:1 ratio (WCAG 2.2 SC 1.4.3)
...
> [VERIFY] Review before adding to the execution log.

<!-- testnux:enrich:end -->
```

Rules:
- Content above `<!-- testnux:enrich:start -->` is immutable from the agent's perspective.
- Each `enrich` run replaces the content **inside** the start/end markers with a fresh set.
- To permanently promote an agent-suggested TC to "human-curated", move it **above** the start marker.
- The markers are added automatically by the first `enrich` run. Add them manually with:
  `testnux enrich <slug>` in v0.1 — it adds the markers without making LLM calls.

## The `[VERIFY]` Confidence Marker Convention

Every LLM-generated cell that requires human confirmation carries a `[VERIFY]` marker.
The marker appears:

- In test-plan.md table cells: `R-42 [VERIFY]`
- As a blockquote footer on every TC: `> [VERIFY] Review before execution.`
- As an inline comment in spec.ts: `// [VERIFY] confirm expected result per plan`

The `testnux report` command renders `[VERIFY]` cells with a visual badge (yellow
background, "VERIFY" label). The badge disappears once a human removes the marker —
that removal is the attestation.

A test pass with zero `[VERIFY]` markers means every cell has been human-reviewed.
A test pass with `[VERIFY]` markers still present can be submitted but should carry
a reviewer note that LLM-generated content was not individually attested.

## Cost Expectations

Estimates assume `claude-sonnet-4-6` pricing (April 2026). Costs vary with DOM complexity
and plan length.

| Stage | Tokens (typical) | Cost per page |
|---|---|---|
| `discover` | 8k–25k in + 4k–8k out | $0.10–$0.35 |
| `plan` | 6k–15k in + 6k–12k out | $0.12–$0.30 |
| `codify` | 8k–20k in + 8k–15k out | $0.18–$0.40 |
| `enrich` (3 passes) | 10k–30k in + 4k–10k out | $0.20–$0.60 |
| **Full pipeline** | — | **$0.60–$1.65** |

For `batch-plan` with 20 pages at 5 pages/agent:
- 4 agent invocations × $3–$8/agent = **$12–$32 total**
- Use `--max-spend $40` as a safe default

Cost guardrails:
- `--max-spend <USD>` — abort before any LLM call if estimated cost exceeds the limit
- `--dry-run` — print the cost estimate without spending
- `--pages-per-agent 3` — reduce batch size to cap per-agent cost

## Wiring `CLAUDE_API_KEY`

All LLM agents require the `CLAUDE_API_KEY` environment variable (Anthropic API key).

```bash
# Option 1: .env.local (recommended for local development)
echo "CLAUDE_API_KEY=sk-ant-..." >> .env.local

# Option 2: shell environment
export CLAUDE_API_KEY=sk-ant-...

# Option 3: inline (not recommended — key appears in shell history)
CLAUDE_API_KEY=sk-ant-... testnux discover https://myapp.com/login
```

The key is read at agent invocation time. It is never written to disk, logged, or
included in test artifacts. The v0.1 stubs work without `CLAUDE_API_KEY` — they exit 0
regardless of whether the key is set.

In v0.2, the agent will check for the key at startup and exit with a clear error if
it is missing, rather than failing mid-operation:

```
Error: CLAUDE_API_KEY is not set.
Set it in .env.local or export it in your shell.
LLM features require an Anthropic API key.
Run without an API key: testnux init <slug>  (scaffolds templates manually)
```

## Rollback and Dry-Run Mechanics

### Dry-run

All LLM commands support `--dry-run`. In dry-run mode:
- The full DOM capture and prompt construction runs.
- The cost estimate is printed.
- No LLM API call is made.
- No files are written.

```bash
testnux discover https://myapp.com/login --dry-run
# Output: estimated tokens, cost estimate, no API call made
```

### Rollback

All file writes are atomic (temp file → rename) in v0.2. If an agent is interrupted
mid-write, the previous file is preserved. To roll back a completed `enrich` run:

```bash
# The enrich section is bounded by markers — just clear the section
# testnux enrich does not modify content above the start marker
# Delete from <!-- testnux:enrich:start --> to <!-- testnux:enrich:end -->
# and re-run when ready
```

For `plan` and `codify`, git is the rollback mechanism:

```bash
git diff testing-log/2026-04-26_login/test-plan.md  # review agent output
git checkout testing-log/2026-04-26_login/test-plan.md  # discard if needed
```

This is why TestNUX artifacts live in your git repo — the full history of
every agent run is tracked, diffable, and reversible.

### Branch workflow

For iterative LLM-agent feedback loops:

```bash
git checkout -b agent/login-plan
testnux discover https://myapp.com/login
testnux plan login
# Review output
# Promote good TCs, delete bad ones, adjust R-IDs
git add testing-log/ && git commit -m "feat(test): login plan from LLM agent, reviewed"
git checkout main && git merge agent/login-plan
```

## Frequently Asked Questions

**Q: Can I run the agents in CI?**
A: Yes, with `--max-spend` as a safety limit. Recommended: gate LLM agent runs on a
separate CI stage triggered manually (not on every PR). Use `testnux report` and
`testnux validate` in the main CI pipeline — those require no API key.

**Q: Do agents overwrite my hand-edited test plans?**
A: `enrich` never modifies content above the `<!-- testnux:enrich:start -->` marker.
`plan` and `codify` write new files; they will not overwrite if the file already exists
unless you pass `--force` (v0.2 flag). In v0.1, the stubs never write test-plan.md.

**Q: Which Claude model is used?**
A: `claude-sonnet-4-6` (cost/speed balance for discover, plan, codify). The `enrich`
design-review pass may use `claude-opus-4-6` for higher-quality accessibility analysis.
Model selection will be configurable in v0.2 via `testnux.config.mjs`.

**Q: Can I use the agents with a locally hosted model?**
A: v0.2 targets the Anthropic API. Local model support (Ollama, LM Studio) is v0.3 scope.
The prompt templates in `src/commands/*.mjs` are model-agnostic — you can adapt them.

**Q: How do I know which TCs were generated by the agent vs written by hand?**
A: Every agent-generated TC ends with `> [VERIFY] ...`. Human-written TCs do not (unless
you add the marker deliberately). The presence of `[VERIFY]` in `testnux report`
output is the visual indicator.
