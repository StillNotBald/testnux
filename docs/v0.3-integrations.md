# v0.3 Integrations

TestNUX ships as a standalone CLI (the default and always-supported path).
v0.3 adds two optional integrations for teams that want tighter editor
tooling: a gstack skill bundle and an MCP server for Claude Code.

---

## Distribution paths

| Path | Availability | Requires | Use when |
|------|-------------|----------|---------|
| **Standalone CLI** | v0.1+ | Node >= 20 | Terminal-first workflows, CI/CD, any editor |
| **gstack skill** | v0.3 | gstack installed | Claude Code + gstack; want `/testnux` as a slash command |
| **MCP server** | v0.3 | `@modelcontextprotocol/sdk` | Claude Code inline tool use; want testnux tools available in chat |

The standalone CLI is always the primary path. The gstack skill and MCP server
are accelerators — they call the same underlying commands.

---

## Decision tree

```
Do you use Claude Code?
├─ No  → Use the standalone CLI.  npx testnux <command>
└─ Yes
   ├─ Do you have gstack installed?
   │   ├─ Yes → Use the gstack skill (/testnux init login)
   │   └─ No  → Use the MCP server (testnux_init tool in Claude Code chat)
   │              OR just use the standalone CLI from a terminal
   └─ Do you want inline tool use in Claude Code chat?
       └─ Yes → Use the MCP server
```

## Option 1: Standalone CLI

The default. No extra dependencies beyond Node >= 20.

```bash
# Install globally
npm install -g testnux

# Or use without installing
npx testnux init login --industry general
npx testnux report testing-log/2026-04-26_login
npx testnux validate testing-log/2026-04-26_login
npx testnux doctor
```

All commands, all industry profiles, full flag support. Output lives in your
repo and is fully auditor-ready.

## Option 2: gstack skill bundle

The gstack skill makes `/testnux` available as a first-class slash command
in any Claude Code session that has gstack installed. The skill handles
preflight checks, routes commands, and surfaces the three-track discipline
conventions (requirements/, sprint-log/, testing-log/).

### Install

```bash
# From the testnux repo root
bash integrations/gstack/testnux/install.sh
```

The script:
1. Detects your gstack skills directory (`~/.claude/skills/gstack/` by default)
2. Symlinks `integrations/gstack/testnux/` into that directory
3. Verifies the SKILL.md is present
4. Optionally runs `gstack-list-skills` to confirm

### Verify

After installation, restart Claude Code and type:

```
/testnux doctor
```

You should see preflight check output. If the skill does not appear, check
that gstack is installed (`~/.claude/skills/gstack/` exists) and restart
Claude Code.

### Usage

```
/testnux init login --industry general
/testnux report testing-log/2026-04-26_login
/testnux validate testing-log/2026-04-26_login --strict
/testnux rtm
/testnux sca init login
/testnux doctor
```

### Failure modes

| Symptom | Cause | Fix |
|---------|-------|-----|
| Skill not found after install | Claude Code did not reload | Restart Claude Code |
| `NOT_INSTALLED` from preflight | testnux CLI not on PATH | `npm install -g testnux` |
| Symlink failed on Windows | Windows requires developer mode for symlinks | install.sh falls back to copy; re-run if SKILL.md is missing at target |
| gstack not found | gstack not installed | Install gstack: https://github.com/stackblitz-labs/gstack |

## Option 3: MCP server for Claude Code

The MCP server exposes testnux commands as tools that Claude Code can call
inline from the chat pane. No terminal switching required. The server
communicates over stdio — Claude Code manages the process lifecycle.

### Prerequisites

```bash
npm install @modelcontextprotocol/sdk
```

`@modelcontextprotocol/sdk` is an optional peer dependency. The MCP server
exits with a friendly error if it is not installed. The standalone CLI and
gstack skill do not require it.

### Install — npx (recommended)

Add to `~/.claude/settings.json` (global) or `.claude/settings.json` (project):

```json
{
  "mcpServers": {
    "testnux": {
      "command": "npx",
      "args": ["-y", "testnux", "mcp"]
    }
  }
}
```

`npx -y` fetches the latest published version of testnux on first run.
Subsequent runs use the npx cache.

### Install — local checkout

If you are working from a local clone:

```json
{
  "mcpServers": {
    "testnux": {
      "command": "node",
      "args": ["/absolute/path/to/testnux/integrations/claude-code-mcp/server.mjs"]
    }
  }
}
```

Replace `/absolute/path/to/testnux` with the actual path to your clone.

### Verify

1. Restart Claude Code after editing settings.json.
2. Open the MCP tool panel (Cmd/Ctrl+Shift+P → "MCP: List Servers").
3. `testnux` should appear as connected.
4. In chat, type: "Run testnux_doctor" — Claude Code will call the tool.

### Available tools

| Tool | Description |
|------|-------------|
| `testnux_init` | Scaffold a test-pass folder |
| `testnux_report` | Generate XLSX + HTML report |
| `testnux_validate` | Lint markdown frontmatter |
| `testnux_doctor` | Preflight checks |
| `testnux_rtm` | Generate/refresh traceability matrix (v0.2+) |
| `testnux_sca` | SCA init / generate / pdf (v0.2+) |

### Failure modes

| Symptom | Cause | Fix |
|---------|-------|-----|
| Server not in MCP list | settings.json not saved or Claude Code not restarted | Save + restart |
| `@modelcontextprotocol/sdk is not installed` | SDK missing | `npm install @modelcontextprotocol/sdk` |
| `rtm command is not yet available` | Using v0.1 CLI | Update to v0.2: `npm install -g testnux@next` |
| Tool calls return error | CWD mismatch | MCP server runs in the project root; confirm Claude Code's CWD matches your project |

## v0.3 / v0.4 roadmap

| Feature | Version | Notes |
|---------|---------|-------|
| gstack skill bundle | v0.3 | Ships in this release |
| MCP server | v0.3 | Ships in this release |
| Browser-coupled discovery via claude-in-chrome MCP | v0.3 | `/testnux discover` uses real Chrome session |
| Cursor adapter | v0.4 | Cursor supports MCP; same server, different config path |
| Continue adapter | v0.4 | Continue.dev supports MCP via the same protocol |
| VS Code extension | v0.4 | Sidebar panel for evidence review; uses the MCP server |
| Hosted SaaS tier | v1.0 | Git-native artifacts + cloud dashboard; MCP server unchanged |

The MCP server interface (tool names and input schemas) follows semver. Breaking
changes to tool signatures require a major version bump. v0.4 adapters build on
top of the v0.3 MCP server — no breaking changes anticipated.
