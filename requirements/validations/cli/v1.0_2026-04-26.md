## 1. Executive Summary

| Metric | Value |
|--------|-------|

| Controls with automated test evidence | 0 (v0.1 — no test suite yet; `npm test` will fail) |
| Controls with manual smoke evidence | 6 |

| Controls with stub/VERIFY status | 5 |
| Overall risk posture | LOW for a local CLI tool; MEDIUM if used in automated pipelines without input sanitization |

**Scope:** This assessment covers the `cli` surface — the `testnux` binary and all
`src/commands/*.mjs` source files invoked from it. The MCP server
(`integrations/claude-code-mcp/server.mjs`) and the gstack skill are adjacent surfaces
with their own SCA scope (see Section 7.3).

**Key findings:**

1. Path traversal mitigation is in place for slug inputs (`init.mjs:134`) but the `validate`
   command accepts an arbitrary folder path — the folder is resolved with `path.resolve`
   which is safe, but downstream file walking should be verified to never escape the
   resolved root. `[VERIFY]`

2. The five LLM-touching commands (`discover`, `plan`, `codify`, `enrich`, `batch-plan`) are
   v0.2 stubs. They do not yet call any external API. When they become functional in v0.2,
   prompt-injection from user-supplied test plans will become a live attack surface.

3. The `uat-log.mjs` HMAC-SHA256 chain implementation uses Node's built-in `crypto` module
   with no third-party HMAC dependency — a positive supply-chain hygiene finding.

**Declined-by-design controls:** 4 (telemetry, auto-update, cloud upload, secret scanning).
See Section 6.
