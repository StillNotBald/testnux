# TestNUX Eval Harness

The eval harness detects **LLM regression** — cases where a prompt or model
change causes `testnux discover`, `plan`, or `codify` to produce materially
worse output on known inputs.

It works by running the three core commands against hand-curated HTML fixtures
and comparing the outputs against **golden files** (expected outputs) using
heuristic scoring. Scores are in [0, 1]. A fixture "passes" when its
overall F1 score is at or above the threshold (default 0.7).

## Why we have an eval set

LLMs are non-deterministic. A prompt wording change, a model upgrade, or a
context window change can silently reduce the quality of generated test cases.

The eval harness catches these regressions by:

1. Running the same fixtures every time.
2. Scoring actual output against hand-curated golden files.
3. Failing CI if any fixture drops below the threshold.

The goldens were written by a human QA engineer — not generated by the LLM —
so we are not grading the model against itself.

## How to run

### Basic run (all fixtures, default threshold 0.7)

```bash
node test/eval/run.mjs
```

### Single fixture

```bash
node test/eval/run.mjs --fixture easy-login
```

### Custom threshold

```bash
node test/eval/run.mjs --threshold 0.8
```

### Dry-run (no API calls — check that fixtures and goldens exist)

```bash
node test/eval/run.mjs --dry-run
```

### Mock mode (CI — no API key needed)

```bash
node test/eval/run.mjs --mock
```

Mock mode loads pre-recorded LLM responses from `test/eval/mocks/<fixture>.json`
instead of calling the Claude API. This lets CI gates run without an API key.

### Live mode (default, requires API key and a fixture server)

Live mode calls the real commands. Because `testnux discover` only accepts
`http://` URLs, you need to serve the fixtures:

```bash
# Terminal 1: serve fixtures
npx serve test/eval/fixtures -p 8080

# Terminal 2: run eval
CLAUDE_API_KEY=sk-ant-... node test/eval/run.mjs
```

## Exit codes

| Code | Meaning |
|------|---------|
| 0    | All fixtures passed at or above threshold |
| 1    | One or more fixtures failed below threshold |

## Threshold semantics

The threshold (default **0.7**) is applied to the **overall F1** of each
fixture, which is the average of:

- `scenarios` F1 — how well `discover` covered the expected TC set
- `plan` F1 — how well `plan` structured the TCs (with R-IDs, [VERIFY], prefix)
- `spec` F1 — how well `codify` generated Playwright code (imports, afterEach,
  [VERIFY] comments, no hardcoded URLs)

### Precision

- **scenarios**: correctly-formed TCs (has Given/When/Then + [VERIFY] + Priority)
  divided by total actual TCs.
- **plan**: same as scenarios, weighted by R-ID format validity (10%),
  [VERIFY] coverage (10%), and TC prefix consistency (10%).
- **spec**: weighted combination of: Playwright import present, test() count,
  [VERIFY] comments, no hardcoded URLs, afterEach hook, accessible locators.

### Recall

- **scenarios** and **plan**: expected TCs found in actual output, matched by
  fuzzy title similarity (Jaccard on word tokens, threshold 0.4). TCs that are
  reworded but cover the same ground still count.
- **spec**: expected test() titles found in actual output.

### Why 0.7?

0.7 is conservative enough that contributors can ship without perfect parity,
while still catching severe regressions (< 50% TC coverage, missing [VERIFY]
markers, broken imports). Raise it to 0.85+ once the pipeline is stable.

## How to add a new fixture

1. **Create the HTML fixture** in `test/eval/fixtures/<slug>.html`.
   Keep it realistic but generic — no company-specific branding.

2. **Write the golden files** in `test/eval/expected/<slug>/`:
   - `scenarios.md` — hand-write the TCs you expect `discover` to produce.
   - `test-plan.md` — hand-write the structured plan you expect `plan` to produce.
   - `spec.ts` — hand-write the first 3–5 `test()` blocks you expect `codify`
     to produce. Truncation is fine; the scorer checks structure, not exhaustiveness.

3. **Add the fixture name** to `KNOWN_FIXTURES` in `test/eval/run.mjs`.

4. **Record a mock** (optional but recommended for CI):
   Run the fixture in live mode, capture the NDJSON output, and save it to
   `test/eval/mocks/<slug>.json`. See `easy-login.json` for the schema.

5. **Run the harness** to confirm your golden passes:
   ```bash
   node test/eval/run.mjs --fixture <slug> --mock
   ```

---

## When to update goldens

Update golden files when:

- **The LLM model is upgraded** (e.g. sonnet-4-6 → sonnet-4-7): run live mode,
  diff the output, review it manually, then replace the golden.
- **The prompt template changes**: same process — diff first, review, replace.
- **A deliberate quality improvement**: if the new output is strictly better
  (more TCs, better standards coverage), update the golden to the new baseline.

**Do NOT auto-generate goldens** from the current LLM output. Goldens must be
written or reviewed by a human — otherwise the eval grades the model against
itself, which defeats the purpose.

## Scoring functions

See `scoring.mjs` for the full implementation. All three functions share the
same return shape:

```ts
{
  precision: number,  // in [0, 1]
  recall:    number,  // in [0, 1]
  f1:        number,  // harmonic mean of precision and recall
  details:   object,  // breakdown for debugging
}
```

### `scoreScenarios(actual, expected)`

Extracts TC blocks from both texts, matches them by Jaccard title similarity,
and checks that each actual TC has Given/When/Then + [VERIFY] + Priority.
Also reports standards recall (OWASP, WCAG, NIST references covered).

### `scorePlan(actual, expected)`

Extends `scoreScenarios` with three plan-specific checks:
- **R-ID format**: every TC row must have `R-XX` or `[] [VERIFY]`.
- **[VERIFY] coverage**: every TC must have a blockquote `[VERIFY]`.
- **TC prefix consistency**: all TCs use the same prefix (e.g. `LOGIN-`).

### `scoreSpec(actual, expected)`

Checks structure, not exact code match:
- Playwright import present
- `test()` count at least 50% of expected
- `// [VERIFY]` comments present
- No hardcoded production URLs (must use `BASE_URL`)
- `test.afterEach()` screenshot hook present
- Prefers `getByRole()` / `getByLabel()` over CSS selectors
