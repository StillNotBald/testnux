# TrunkNuX â€” Requirements

> **Project:** TrunkNuX (renamed from testnux 2026-04-27, v0.2.2)
> **Repo:** github.com/StillNotBald/trunknux
> **Owner:** Chu Ling

This document is the canonical source-of-truth for what TrunkNuX *claims to do*. Every R-XX entry is testable, every test case (TC-ID in testing-log/) must reference at least one R-XX, and every signoff record must attest to specific R-IDs.

This is a **dogfood document** â€” TrunkNuX is the tool that produces audit-defensible evidence chains, so the project itself uses the same discipline. The requirements here flow into:
- `requirements/TRACEABILITY.md` (generated by `trunknux rtm`)
- `testing-log/<date>_trunknux-cli/test-plan.md` (TC matrix)
- `sca/trunknux-cli.md` (Security Control Assessment)
- `uat-log.jsonl` (HMAC-chained signoff ledger)
- `oscal-assessment.json` (NIST OSCAL 1.1.2 export)

## R-28 â€” Three-track folder structure

**Acceptance:**

- `trunknux rtm` reads from `requirements/` and cross-references to `testing-log/`
- `trunknux br rtm` writes to `requirements/` and is not mixed into `testing-log/`
- Template scaffolds include a `README.md` stub pointing at the track convention
- Deviating from the three-track layout causes `trunknux validate --strict` to warn
