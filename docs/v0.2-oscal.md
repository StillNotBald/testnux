# OSCAL Export — TestNUX v0.2

TestNUX v0.2 emits NIST OSCAL 1.1.2 `assessment-results` JSON alongside every
Security Control Assessment (SCA) markdown document. No Python or external dependencies
are required; TestNUX generates conformant OSCAL directly in Node.js.

---

## Why OSCAL?

**FedRAMP RFC-0024** mandates machine-readable OSCAL packages for all FedRAMP
authorization deliverables by **September 2026**. Federal agencies and federal-adjacent
customers (state governments, regulated contractors, financial institutions filing FFIEC
or NYDFS reports) increasingly require audit evidence in OSCAL format rather than or in
addition to PDF/XLSX.

OSCAL (Open Security Controls Assessment Language) is the NIST standard for representing
security control catalogs, system security plans, assessment plans, and assessment results
in machine-readable JSON or XML. OSCAL-formatted evidence is:

- **Queryable** — RegScale, Xacta, IBM OpenPages, and GovReady-Q can ingest it directly.
- **Comparable** — diff two assessments across quarters without reading prose.
- **Auditor-facing** — FedRAMP reviewers accept OSCAL packages in place of manual spreadsheets.
- **Interoperable** — converts to STIX, CSAF, and other compliance formats via open toolchains.

## Mapping: SCA Sections → OSCAL Elements

| SCA Section | SCA Field | OSCAL Element |
|---|---|---|
| Frontmatter | `surface`, `version` | `assessment-results.metadata.title`, `metadata.version` |
| Frontmatter | `published` | `metadata.published` |
| Frontmatter | `signOff[]` | `metadata.parties[]`, `metadata.roles[]`, `results[0].responsible-parties[]` |
| Controls table | Control row (AC-2, IA-5, …) | `results[0].reviewed-controls.control-selections[0].include-controls[]` |
| Controls table | Status: PASS / FAIL / PARTIAL | `results[0].findings[].target.status.state` (satisfied / not-satisfied) |
| Controls table | Status: N/A | `results[0].findings[].target.status.state` (not-applicable) |
| Evidence links | Href, description | `results[0].findings[].links[]` (rel: "evidence") |
| Bullet findings | Observation text | `results[0].findings[].description` |
| Declined-by-design section | controlId, reason, approvedBy | `results[0].risks[]` with `risk-status: deviation-requested` |
| Declined approval | approvedBy, approvedDate | `risks[].risk-log.entries[]` |

## IBM Compliance Trestle

[IBM Compliance Trestle](https://ibm.github.io/compliance-trestle/) is the canonical
Python toolchain for authoring, validating, and transforming OSCAL documents. The
TestNUX OSCAL emitter produces output that is **Trestle-compatible** without
requiring Python to be installed.

| | Trestle | TestNUX OSCAL |
|---|---|---|
| Language | Python 3.9+ | Node.js 20+ |
| Install | `pip install compliance-trestle` | No extra install (built in) |
| Generates | SSP, CAP, SAR, POA&M | Assessment Results (`assessment-results`) |
| Validates | Full OSCAL schema | Required-field subset (fast, CI-safe) |
| Converts | OSCAL ↔ Excel, OSCAL ↔ Markdown | Markdown SCA → OSCAL JSON |

The two tools are complementary: TestNUX generates the assessment-results document
from your SCA markdown; Trestle validates, transforms, and submits to FedRAMP/agency systems.

**Typical combined workflow:**

```bash
# 1. Generate OSCAL from your SCA markdown
testnux sca oscal login --validate

# 2. Import into Trestle for full validation and SSP linkage
trestle import -f requirements/validations/login/v0.1.oscal.json \
               -o login-assessment-results

# 3. Run Trestle's full schema validation
trestle validate -t assessment-results -n login-assessment-results

# 4. Use Trestle to generate FedRAMP SAR Word document
trestle author docs assemble ...
```

## Validation Command

```bash
# Emit OSCAL and run the built-in schema check
testnux sca oscal login --validate

# Dry-run: parse and validate without writing the file
testnux sca oscal login --validate --dry-run

# Write to a custom output directory
testnux sca oscal login --out ./oscal-packages/

# JSON output (for CI integration)
testnux --json sca oscal login --validate
```

The `--validate` flag runs a minimal schema check covering:

- Required fields present (`uuid`, `metadata`, `results`)
- All UUIDs match RFC-4122 format (`/^[0-9a-f]{8}-[0-9a-f]{4}-...-[0-9a-f]{12}$/i`)
- All dates are valid ISO-8601
- `oscal-version` is present
- `results` array is non-empty

Exit codes:

| Code | Meaning |
|------|---------|
| 0 | Success — OSCAL written (or validated in dry-run) |
| 1 | OSCAL validation failed |
| 2 | Missing or invalid input (no SCA file found) |
| 3 | Parse error (malformed SCA markdown) |

## Notes for Implementers

- The OSCAL UUIDs emitted by TestNUX are deterministic (seeded from surface + version).
  This makes diffs readable — the same control always gets the same UUID across regenerations.
  In v0.2, swap the hash-based UUID generator in `src/lib/oscal.mjs` with `uuid` package v5
  for full RFC-4122 compliance.

- The `import-ap` href (`#assessment-plan-<surface>`) is a placeholder. In production, replace
  with the actual Assessment Plan UUID from your SSP package before submitting to FedRAMP.

- TestNUX does not yet emit `assessment-plan` or `system-security-plan` OSCAL documents.
  Those are v0.3 scope. Use Trestle to author those components and link back to the
  TestNUX `assessment-results` UUID.
