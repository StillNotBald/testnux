# TestNUX v0.3 — Per-Environment Passes + Visual Regression

> **Status:** Per-environment scaffold is fully functional. Visual regression is a v0.3 stub — install `pixelmatch` to enable comparison.

## Why Per-Environment Passes Matter

v0.1 creates one test pass per feature (e.g., `2026-04-26_login-23-tc/`).
v0.3 creates one test pass per (feature × environment):

```
testing-log/
  2026-04-26_login-staging/
  2026-04-26_login-prod/
  2026-04-26_login-local/
```

**Config drift** — differences in environment variables, feature flags, network policies, or data seeding — is the most common cause of "passes on staging, fails on prod" regressions. Per-env passes make this drift:

1. **Visible** — you have dated, immutable records of what was tested where
2. **Comparable** — `testnux compare` diffs TC results across environments
3. **Auditable** — regulators and auditors can see which environment was tested and when

### Common Config Drift Patterns

| Drift Type | Example | Detected by compare? |
|------------|---------|---------------------|
| Feature flag off in prod | `ENABLE_MFA=true` staging, unset prod | Yes — TC fails in prod |
| Rate limiting difference | 100 req/min staging, 10 req/min prod | Yes — TC flaky in prod |
| Auth provider config | Test OIDC client in staging, real client prod | Yes |
| Static asset CDN | Local file staging, CDN URL prod | Visual regression |
| Date-based content | "Today's balance" value differs by env | Mask with visual.maskSelectors |

## The Compare Workflow

### Scaffold both environment passes

```bash
testnux run login --env staging
testnux run login --env prod
```

Creates:
- `testing-log/<date>_login-staging/`
- `testing-log/<date>_login-prod/`

### Run tests in each environment

```bash
# Point your test runner at staging
APP_URL=https://staging.example.com npx playwright test testing-log/<date>_login-staging/spec.ts

# Point at prod
APP_URL=https://example.com npx playwright test testing-log/<date>_login-prod/spec.ts
```

Fill in `execution-log.md` in each folder (or let `testnux report` generate it from Playwright output).

### Compare

```bash
testnux compare login staging prod
```

Output:

```
| TC-ID    | staging status | prod status | delta              |
|----------|---------------|-------------|---------------------|
| LOGIN-01 | PASS          | PASS        | ✓                  |
| LOGIN-15 | PASS          | FAIL        | ⚠ REGRESSION       |
| LOGIN-16 | PASS          | PASS        | ✓                  |
| LOGIN-17 | FAIL          | PASS        | ℹ PROMOTION        |
```

**Delta values:**
- `✓ MATCH` — same result in both environments
- `⚠ REGRESSION` — passes in env-A but fails in env-B (typical: staging→prod regression)
- `ℹ PROMOTION` — fails in env-A but passes in env-B (atypical: investigate env-A config)
- `⚡ DIVERGE` — different non-PASS/FAIL statuses (e.g. SKIP vs BLOCKED)
- `only in <env>` — TC exists in one pass but not the other

### CI Gate Use Case

Add this to your PR pipeline:

```yaml
# .github/workflows/uat.yml
- name: Compare staging vs prod
  run: |
    npx testnux compare login staging prod --json | \
      jq -e '[.diff[] | select(.delta == "REGRESSION")] | length == 0'
```

This fails the PR if any TC that passed in staging is failing in prod.

## Visual Regression

### What It Does

After each TC, capture a full-page screenshot. Compare to the baseline (captured after the last approved release). Flag TCs where the pixel diff exceeds the configured threshold.

This catches:
- Layout regressions (element moved, overlapping content)
- Style regressions (color, font, spacing changes after a CSS refactor)
- Missing content (component not rendering)

It does NOT replace functional testing — a pixel diff of 0 means nothing if the component has the wrong data.

### Install

```bash
npm install pixelmatch pngjs
```

pixelmatch is MIT licensed and has zero runtime dependencies. It is an optional peer dependency of testnux — visual regression is disabled gracefully if it is not installed.

### Capture Baseline

```bash
testnux visual baseline login
```

Captures full-page screenshots for every TC in the most recent `login` test pass.
Stored at: `testing-log/<date>_login/visual-baseline/<TC-ID>.png`

**Baseline strategy:**

| Strategy | When to use |
|----------|-------------|
| **Per release** | Capture baseline after each approved release. Re-baseline only on intentional UI changes. Best for stable, release-gated products. |
| **Per branch** | Capture baseline from the base branch. Compare on PR. Best for rapid-iteration products. Requires CI image storage. |
| **Per environment** | Capture prod baseline. Compare staging to it. Best for detecting env-specific rendering bugs. |

### Compare Against Baseline

```bash
testnux visual compare login
# Flag-only mode (default — no exit code change)

testnux visual compare login --strict
# Fail with exit code 1 if any TC exceeds threshold
```

Diffs stored at: `testing-log/<date>_login/visual-diff/<TC-ID>-diff.png`

Output:

```
| TC-ID    | Status    | Diff %  | Above Threshold |
|----------|-----------|---------|-----------------|
| LOGIN-01 | PASS      | 0.12%   | no              |
| LOGIN-15 | FAIL      | 8.43%   | YES             |
```

### Pixel-Diff Threshold Tuning

The default threshold is **5% of pixels** (0.05). Configure in `testnux.config.mjs`:

```js
export default {
  visual: {
    diffThreshold: 0.05,   // 5% — adjust per surface
    fullPage: true,
    maskSelectors: [],
  }
}
```

**Tuning guidance:**

| Surface | Recommended threshold | Reason |
|---------|-----------------------|--------|
| Static marketing pages | 0.01 (1%) | No dynamic content; regressions are real |
| Dashboard with live data | 0.10 (10%) | Data values change; focus on layout |
| Charts / graphs | 0.15 (15%) | Pixel rendering varies by OS/font |
| Forms | 0.02 (2%) | Layout matters; no live data |

Start at 5% and tighten once you have baseline noise data from real runs.

### False-Positive Mitigation

Visual regression generates false positives when non-visual changes affect screenshot pixels. Three strategies:

#### 1. Stable Selectors

Use `data-testid` attributes instead of CSS selectors for element targeting. Test IDs are stable across refactors; class names change.

#### 2. Deterministic Test Data

Seed the database with fixed test data before capturing screenshots. Date/time values, counters, and user-specific content all vary between runs.

For date-based content, mock `Date.now()` in your Playwright spec:
```ts
await page.addInitScript(() => {
  Date.now = () => new Date('2026-04-26T00:00:00Z').getTime();
});
```

#### 3. Mask Dynamic Regions

Configure `visual.maskSelectors` to exclude regions with expected pixel variance:

```js
export default {
  visual: {
    diffThreshold: 0.05,
    maskSelectors: [
      '[data-testid="live-clock"]',
      '[data-testid="animated-counter"]',
      '.recharts-tooltip',        // charts tooltips flicker
    ],
  }
}
```

Masked regions are painted solid gray before diffing — any change inside the mask is ignored.

#### 4. Cross-Platform Consistency

Font rendering, anti-aliasing, and scroll behavior differ between operating systems. For CI:

- Pin the OS in your CI runner (e.g. `ubuntu-22.04`, not `ubuntu-latest`)
- Use the same Playwright version and Chromium build locally and in CI
- Run visual regression only on Linux CI, not macOS, to avoid font diff noise

## Summary: v0.3 Environment Commands

| Command | What it does |
|---------|-------------|
| `testnux run <slug> --env staging` | Scaffold `<date>_<slug>-staging/` |
| `testnux run <slug> --env prod` | Scaffold `<date>_<slug>-prod/` |
| `testnux compare <slug> staging prod` | Diff TC results; flag regressions |
| `testnux visual baseline <slug>` | Capture baseline screenshots |
| `testnux visual compare <slug>` | Compare to baseline; flag diffs |
| `testnux visual compare <slug> --strict` | Same, fail on diff |
