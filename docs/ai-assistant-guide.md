## What TrunkNuX is

A CLI tool that produces audit-defensible evidence chains for regulated software. It covers five stages of the testing-to-audit journey:

```
   ENGINEER         QA            COMPLIANCE        AUDITOR
   ─────────       ─────          ──────────       ─────────
   test plan   →   reports    →   signoffs    →   evidence
   + spec          XLSX/HTML      HMAC PDF         package
   + evidence      RTM            SCA + OSCAL      hash-chain
                                                   verifiable
```

OSS (Apache 2.0). Git-native. All artifacts live in your repo. No vendor cloud. Auditors can verify the hash chain independently of the tool.

The product is **deterministic at the core, AI-accelerated at the edges**. The `report` command needs no LLM and produces the same output every run. AI features (`discover`, `plan`, `codify`, `enrich`, `batch-plan`) are opt-in accelerators with explicit cost gates.

## The discipline (non-negotiable rules)

These rules are what makes the evidence audit-defensible. Violating them turns the package into theater.

### Rule 1: `[VERIFY]` markers gate all LLM-generated content

Every cell, sentence, or test case generated by an LLM (via `discover`, `plan`, `codify`, `enrich`, `batch-plan`, or `sign --justify-with-llm`) renders with a `[VERIFY]` prefix or comment. That marker means: **no human has attested to this content yet.**

**DO NOT** remove a `[VERIFY]` marker without:
1. Reading the underlying content carefully
2. Validating it is correct for this specific project / control
3. Filling any gaps the LLM left
4. Running any related tests if applicable

**DO NOT** auto-remove `[VERIFY]` markers in batch operations (no `sed -i 's/\[VERIFY\]//g'`). Each removal is a human attestation. Treat it as one.

If you (the AI) generate content for a TrunkNuX artifact, **always** add `[VERIFY]` markers to each cell you produce. Do not assume the user has verified your output.

### Rule 2: HMAC chain integrity

The signoff ledger (`uat-log.jsonl`) is HMAC-chained. Each entry's signature depends on the previous entry. Tampering breaks the chain.

**DO NOT** edit `uat-log.jsonl` directly. Always use `trunknux sign` so the chain stays valid.

**DO NOT** regenerate the chain from scratch unless you are intentionally revoking previous attestations. Even then, prefer `trunknux sign --revoke` which appends a revocation entry instead of removing.

If you suspect a chain is broken, run `trunknux sign --verify <surface>` to confirm before doing anything else.

### Rule 3: Three-track structure

Every regulated project should organize artifacts in three parallel tracks:

```
requirements/                    ← what you said you'd build
  REQUIREMENTS.md                  R-XX entries (h2 headings: ## R-01 — title)
  TRACEABILITY.md                  generated by `trunknux rtm`

sprint-log/<date>_<feature>/     ← what was built per sprint
  SPRINT_SUMMARY.md

testing-log/<date>_<page>/       ← what was tested per test pass
  test-plan.md
  spec.ts
  execution-log.md
  evidence/
  *.xlsx
  *.html
  uat-log.jsonl
```

Date-prefix the testing-log and sprint-log folders. This creates audit snapshots: every engagement has a clear record of what was tested on what date.

R-IDs use sequential format (`R-01`, `R-58`). They live in `requirements/REQUIREMENTS.md` as h2 headings. The parser regex matches `R-` followed by 2-4 digits with optional letter suffix. Group prefixes like `R-CMD-01` do NOT parse; use `R-01` and add a section header for the group.

### Rule 4: Honesty over polish

The SCA, signoff ledger, and OSCAL export reflect what is actually true. If a control is partially implemented, mark it PARTIAL. If a test was skipped, log SKIPPED with reason. Do not pad coverage numbers. Do not invent attestations.

The `[VERIFY]` discipline assumes the human reviewer is honest about what they verified. If the AI is asked to "make it look good," the answer is no.

Auditors catch dishonesty. The product's value depends on trustworthy artifacts.

### Rule 5: Cost gates on LLM commands

`discover`, `plan`, `codify`, `enrich`, `batch-plan` all call Claude API. They cost real money.

**Always** run `--dry-run` first to see the cost estimate before the actual call. Use `--max-spend <usd>` on batch operations to cap exposure. Default rough costs (Sonnet-class):

- `discover` per page: ~$0.10
- `plan` per page: ~$0.15
- `codify` per page: ~$0.20
- `enrich` per page (3 passes): ~$0.30-$0.50

Cap a multi-page batch with `--max-spend 5.00` and the run aborts when cumulative spend hits the cap.

## Common workflows (recipe book)

### Recipe 1: Set up a new project for TrunkNuX

```bash
npm install -g trunknux
mkdir -p requirements testing-log sprint-log

# Author your first requirements (h2 headings, R-NN format)
$EDITOR requirements/REQUIREMENTS.md

# Run preflight
trunknux doctor
```

Each R-ID needs: heading (`## R-01 — title`), `**Status:** DONE | PARTIAL | PLANNED`, `**Description:**`, `**Acceptance:**`, `**Test coverage:**`. See trunknux's own `requirements/REQUIREMENTS.md` for a worked example with 58 R-IDs.

### Recipe 2: Per-page test pass

```bash
trunknux init my-page --industry general
# creates testing-log/<date>_my-page/

# Edit test-plan.md to enumerate TC matrix referencing R-IDs
# Edit spec.ts to implement the test cases

npm test       # or: npx playwright test testing-log/<date>_my-page/spec.ts

# Capture results in execution-log.md (narrative + per-TC PASS/FAIL)

trunknux validate testing-log/<date>_my-page/   # lint frontmatter (always before report)
trunknux report testing-log/<date>_my-page/     # produces XLSX + HTML
```

### Recipe 3: Generate SCA + signoff + OSCAL for the surface

```bash
trunknux sca init my-page                       # scaffold sca/my-page.md
trunknux sca generate my-page                   # fill from test results

trunknux sign testing-log/<date>_my-page/       # interactive: per-TC reviewer prompts
                                                # writes uat-log.jsonl with HMAC chain

trunknux sca pdf my-page                        # PDF (needs puppeteer-core: npm install puppeteer-core)
trunknux sign pdf testing-log/<date>_my-page/   # signoff PDF (same dep)

trunknux sca oscal my-page                      # NIST OSCAL JSON
                                                # output: requirements/validations/my-page/v1.0.oscal.json

trunknux rtm                                    # regenerate TRACEABILITY.md
```

The signoff command is **interactive**. It prompts for reviewer name, role, status, justification per TC. If you are an AI assistant helping a user, do not run `trunknux sign` non-interactively without `--reject` or `--revoke` flags; the human attestation is the whole point.

### Recipe 4: AI-augmented authoring (opt-in)

```bash
export CLAUDE_API_KEY=sk-ant-...
npm install @anthropic-ai/sdk

trunknux discover https://my-app.com/login --slug login --dry-run    # see cost first
trunknux discover https://my-app.com/login --slug login              # actual call
# outputs: scenarios.md with [VERIFY]-marked TC suggestions

trunknux plan login --dry-run
trunknux plan login
# outputs: testing-log/<date>_login/test-plan.md (all cells [VERIFY])

# HUMAN ATTEST PASS: read each [VERIFY] cell, validate, fix, remove the marker

trunknux codify login --safe       # writes spec.generated.ts if spec.ts already exists
trunknux enrich login              # three append-only passes (design-review/qa-structural/graph-context)

trunknux batch-plan --pages "login,register,checkout" --max-spend 5.00 --pages-per-agent 3
```

`--safe` mode on `codify` writes to `spec.generated.ts` instead of overwriting an existing `spec.ts`. Use this when humans have already hand-edited the spec.

`enrich` is marker-bounded. Re-running it replaces the marker block but preserves human edits OUTSIDE the markers.

### Recipe 5: Per-environment regression

```bash
trunknux run my-page --env staging        # creates testing-log/<date>_my-page-staging/
trunknux run my-page --env prod           # creates testing-log/<date>_my-page-prod/

trunknux compare my-page staging prod
# verdicts per TC: MATCH / PROMOTION / REGRESSION / DIVERGE / MISSING-A / MISSING-B
# CI gate: trunknux compare my-page staging prod --threshold 0   (exits 1 on any regression)
```

### Recipe 6: Visual regression

```bash
trunknux visual baseline my-page                 # capture full-page screenshots
trunknux visual compare my-page --threshold 0.05 # pixelmatch diff
trunknux visual compare my-page --strict         # CI gate (exits 2 on any diff above threshold)
```

If `pixelmatch` is not installed, the command degrades to screenshot-only mode and prints an install hint.

### Recipe 7: Stale-check and re-attestation

```bash
trunknux sign stale-check testing-log/<date>_my-page/ --threshold 90d
# flags entries older than 90 days

trunknux sign stale-check ... --strict           # CI gate (exits 1 on any stale)

# To re-attest a stale TC, run trunknux sign and re-do the per-TC prompts.
```

## Common pitfalls AI assistants make

### Pitfall 1: Inventing R-IDs

`trunknux rtm` only includes R-IDs that exist in `requirements/REQUIREMENTS.md`. Referencing R-99 in a test plan when R-99 isn't in REQUIREMENTS.md silently drops the row.

**DO:** grep REQUIREMENTS.md before referencing an R-ID.
**DO NOT:** invent R-IDs to make a test plan look complete.

### Pitfall 2: Wrong R-ID heading format

The parser regex is `/^#{1,3}\s+(R-\d{2,4}[A-Z]?)\s*(?:[-—]+\s*)?(.+)?$/`. It accepts h1/h2/h3 headings, with `R-` followed directly by 2-4 digits.

**Works:** `## R-01 — Title`, `### R-99A Title`, `# R-1234 — Title`
**Does NOT work:** `## R-CMD-01 — Title`, `## R-FOO Title`, `**R-01**`

If you want grouping, use section headers ABOVE the R-ID headings. Don't embed group prefixes in the ID.

### Pitfall 3: Removing `[VERIFY]` in bulk

Each `[VERIFY]` is a deliberate gate. Bulk removal forfeits the audit-defensibility claim.

### Pitfall 4: Editing `uat-log.jsonl` directly

The HMAC chain breaks the moment you touch the JSONL with a text editor. Always use `trunknux sign` to append.

To delete or revoke an attestation, use `trunknux sign --revoke --tc <TC-ID> --role <role>`. This appends a revocation entry without breaking the chain.

### Pitfall 5: Skipping `trunknux validate` before `trunknux report`

If frontmatter is malformed, the report renders with broken cross-references. Always validate first.

### Pitfall 6: Confusing surface names with folder paths

`trunknux sca init <surface>` takes a surface name (e.g., `login-page`).
`trunknux sign <folder>` takes a folder path (e.g., `testing-log/2026-04-27_login-page`).

Read `--help` carefully. Do not pass folder paths where a surface name is expected.

### Pitfall 7: Signing TCs without running the test

If you sign off on a TC that hasn't actually been tested, you've created a false attestation. The signoff PDF will say PASS but the underlying evidence is empty.

**Always** run the test suite (`npm test` / `npx playwright test`) before signing. The `execution-log.md` should reflect a real run, not a placeholder.

### Pitfall 8: Forgetting `--dry-run` on LLM commands

LLM commands cost money. `--dry-run` returns the prompt + cost estimate without calling the API. Use it on every first run of a new command or page.

### Pitfall 9: Leaking forbidden strings into committed artifacts

If your project has a list of strings that must never appear in OSS commits (proprietary project names, internal acronyms, etc.), check ALL generated artifacts before committing, not just source code. The HTML report, XLSX, JSONL, and OSCAL JSON can all contain text from your test-plan.md.

`trunknux validate` does not currently check this. Add a project-specific grep step to your CI.

### Pitfall 10: Not updating REQUIREMENTS.md when adding features

The RTM is only as accurate as REQUIREMENTS.md. If you ship a new feature without an R-ID, it shows up nowhere in the audit chain.

**Before merging any feature PR:** add an R-ID to REQUIREMENTS.md. Treat REQUIREMENTS.md as code.

## What TrunkNuX is NOT

Do **not** use TrunkNuX for:

- Personal task management (use Taskwarrior or Org-mode)
- Replacing your test runner (TrunkNuX wraps; vitest / playwright run)
- Replacing your CI (TrunkNuX produces evidence; CI runs the tests)
- Replacing a vendor GRC platform's analytics (consume TrunkNuX outputs in Vanta / Drata / Secureframe / RegScale via OSCAL ingest)

Do **not** claim audit-defensibility:

- Without running the actual tests
- Without human review of every `[VERIFY]` cell
- Without a valid HMAC chain (run `--verify` to confirm)
- Without standards alignment specific to your industry

## File-edit conventions for AI assistants

When you (an AI assistant) generate or modify TrunkNuX-managed files in a user's project, follow these patterns:

| File | What to do | What NOT to do |
|---|---|---|
| `requirements/REQUIREMENTS.md` | Add R-IDs as h2 headings (`## R-NN — title`) with the four required fields | Do not invent R-IDs that aren't documented; do not skip the four fields |
| `requirements/TRACEABILITY.md` | Re-run `trunknux rtm` to regenerate; only edit Notes columns inside `<!-- trunknux:row R-XX begin/end -->` markers | Do not edit other columns by hand; they're regenerated |
| `testing-log/<date>_<slug>/test-plan.md` | TC matrix in the table format, then per-TC G/W/T sections; reference real R-IDs from REQUIREMENTS.md | Do not invent R-IDs or TC-IDs that don't tie to real tests |
| `testing-log/<date>_<slug>/spec.ts` | Real Playwright code that implements the TCs; preserve XFF isolation, `form.requestSubmit()`, `afterEach` evidence hooks if codified by the LLM | Do not write tests that always pass; do not skip assertion specificity |
| `testing-log/<date>_<slug>/execution-log.md` | Real test run summary with timestamps, per-TC PASS/FAIL/SKIP, root-cause notes for failures | Do not fabricate execution results; do not paste raw test output without synthesis |
| `testing-log/<date>_<slug>/uat-log.jsonl` | Always use `trunknux sign` to append; never edit by hand | Editing breaks the HMAC chain |
| `sca/<surface>.md` | Generated by `trunknux sca generate`, then human-attested; honest gaps in the Gaps section | Do not pad coverage; do not hide gaps |
| `requirements/validations/<surface>/*.oscal.json` | Generated by `trunknux sca oscal`; do not hand-edit | Editing breaks OSCAL schema validation |

If you (the AI) write a file that TrunkNuX would normally generate, mark every cell with `[VERIFY]` so the human reviewer knows you wrote it.
